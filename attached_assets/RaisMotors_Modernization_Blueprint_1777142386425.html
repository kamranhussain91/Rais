Architecture Problems Identified

God Controller — Single 1,400-line Admin.php
All 60+ business operations are stuffed into one controller. This makes the code impossible to maintain, test, or scale. A sale of a bike involves 15+ database operations all done inline in the controller with no transaction safety.
No Separation of Concerns
Business logic (stock updates, installment calculations, ledger entries) is mixed with HTTP routing logic inside the same methods. If one step fails mid-sale, the database may be left in a corrupted half-state.
Inconsistent Session Checks
Some methods check session->userdata('id') while the constructor checks session->userdata('admin_id'). This inconsistency means some pages may be accessible without a valid session.
No Database Transactions
The cash/installment sale process involves 7–10 separate INSERT/UPDATE operations. If the browser closes or PHP crashes midway, data will be inconsistent. No $this->db->trans_start() is used anywhere.
2 Security Vulnerabilities

Critical security issues found in the current system that must be fixed before or during modernization.

🚨 CRITICAL: MD5 Password Hashing
Your login system uses md5($password) to hash passwords. MD5 was broken for security purposes over a decade ago — rainbow table attacks can crack MD5 hashes instantly.

Found in database: Admin password hash is ff4c16a0eaa2de07762530224d372a9e — this is a standard MD5 hash that can be cracked in seconds using free online tools.

Fix: Replace with password_hash($password, PASSWORD_BCRYPT) and verify with password_verify().
🚨 CRITICAL: No CSRF Protection
Forms do not include CSRF tokens. An attacker could craft a malicious page that submits forms to your system on behalf of a logged-in admin (Cross-Site Request Forgery). This could create fake sales, delete records, or change settings.

Fix: Enable $config['csrf_protection'] = TRUE; in CodeIgniter config and add <?php echo form_open('admin/adminlogin'); ?> to all forms.
⚠️ Raw $_GET Parameters Without Validation
Multiple methods use $lid = $_GET['pid'] directly without type-casting or validation. This could expose integer overflow attacks or unexpected behavior. Example: /admin/editproduct?pid=../../etc/passwd

Fix: Use (int)$this->input->get('pid') or CI's input filtering everywhere.
⚠️ Login Error Reveals System Info
Failed login echoes echo "error "; — a raw string response with no redirect or proper error page. Error messages should never confirm specific failure reasons in production.
ℹ️ No Login Rate Limiting
The login endpoint has no brute-force protection. An attacker can make unlimited password attempts. Fix: Add a failed-attempt counter in session/database and lock out after 5 failures.
ℹ️ .sftp-config.json Exposed in Web Root
A file sftp-config.json is present in public_html and may contain server credentials. This file must be deleted immediately or blocked via .htaccess.

9 Deployment Strategy (Hostinger)

Since you are on Hostinger shared hosting, the deployment stays on the same server. Here is the exact step-by-step process.

⚠️ Important: Never deploy directly to your live site. Always test on a staging copy first. We will create a /staging subfolder to test the new version alongside your live system.
Backup Everything (Before Starting)
Export database from Hostinger phpMyAdmin → Export → SQL format. Download entire public_html via FileManager or SFTP. Store safely on your local computer. Your app also has a database_backup() function — run it once now.
Create Staging Environment
In Hostinger File Manager, create public_html/v2/ folder. Upload the new CodeIgniter 4 project there. Create a v2_test database in Hostinger MySQL Manager. Import your existing SQL dump into this test database. Test everything at yourdomain.com/v2/
Configure Environment Variables
Create .env file in your CI4 project root (this file must NEVER be committed to GitHub or uploaded publicly):

Set Up GitHub Repository
# On your local computer git init rais-motors-v2 cd rais-motors-v2 git add . git commit -m "Initial CodeIgniter 4 setup" git remote add origin https://github.com/YOUR_USER/rais-motors.git git push -u origin main # IMPORTANT: Add to .gitignore echo ".env" >> .gitignore echo "writable/" >> .gitignore echo "vendor/" >> .gitignore
Deploy to Hostinger via Git Pull
Hostinger supports Git deployment. In Hostinger hPanel → Git → Connect your repository. Set deploy path to public_html. On every push to main branch, Hostinger auto-pulls the new files. The database is never touched by deployments — only PHP files update.
Go Live — Swap Traffic
When staging is fully tested: rename old public_html to public_html_old_backup. Move v2 contents to public_html. Update database credentials in .env to point to your live database. Done — zero downtime.
